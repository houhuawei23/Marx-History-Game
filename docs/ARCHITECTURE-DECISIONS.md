# 架构决策记录 (ADR)

## 文档概述

本文档记录《资本：轮回与破局》项目中的重要架构决策（Architecture Decision Records），包含决策背景、替代方案分析、最终决策及后果。

## ADR-001: 采用纯前端 ES Modules 架构

### 状态
已采纳

### 背景
项目初始阶段需要确定 JavaScript 代码组织方式。当时面临的选择是：是否引入打包工具（如 Webpack、Rollup）。

### 决策
采用原生 ES Modules，不引入任何打包工具。

### 替代方案

| 方案 | 优点 | 缺点 |
|-----|------|------|
| ES Modules（采纳） | 无需构建、原生支持、调试友好、Tree-shaking | 需 HTTP 服务器运行 |
| Webpack | 生态丰富、功能强大 | 配置复杂、构建时间长 |
| Rollup | 输出干净、适合库 | 配置学习曲线 |
| 直接引入（IIFE） | 简单直接 | 无模块化、难以维护 |

### 理由
1. 游戏逻辑相对简单，不需要复杂的模块化处理
2. 内容创作者需要能够直接修改 JS 数据文件
3. 开发调试时无需构建步骤，提高效率
4. 配合本地 HTTP 服务器（`python -m http.server`）即可运行

### 后果
- **正面**：开发体验流畅，调试直观
- **负面**：用户必须通过 HTTP 服务器访问（已通过桌面应用解决）

## ADR-002: YAML 配置文件外置

### 状态
已采纳

### 背景
游戏包含大量内容数据（事件、成就、碎片），需要确定数据存储格式。

### 决策
将所有内容数据存储在 YAML 配置文件中，JavaScript 代码仅包含逻辑。

### 替代方案

| 方案 | 优点 | 缺点 |
|-----|------|------|
| YAML（采纳） | 内容友好、支持注释、层级清晰 | 需解析、类型安全弱 |
| JSON | 原生支持、解析快 | 不支持注释、层级冗余 |
| JavaScript 对象 | 灵活、可直接执行 | 逻辑混淆、版本控制困难 |

### 理由
1. **内容创作者友好**：YAML 支持注释，便于说明
2. **数据逻辑分离**：设计师可改内容，开发者改逻辑
3. **版本控制友好**：纯文本格式，diff 清晰
4. **层级清晰**：适合事件这种树状结构数据

### 后果
- **正面**：内容维护成本降低，非开发者可参与内容创作
- **负面**：运行时需解析 YAML（引入 js-yaml 依赖）

## ADR-003: Web Audio API 程序化音频

### 状态
已采纳

### 背景
游戏需要背景音乐和多种音效。传统方案是预制音频文件。

### 决策
背景音乐使用 HTML5 Audio，音效使用 Web Audio API 程序化合成。

### 替代方案

| 方案 | 优点 | 缺点 |
|-----|------|------|
| Web Audio API（采纳） | 包体积小、灵活可编程 | 实现复杂度高 |
| 预制 MP3/WAV | 实现简单 | 包体积大（~50MB+） |
| Howler.js | 封装良好、兼容性好 | 增加依赖 |

### 理由
1. **包体积优化**：程序化合成无需预制音频，节省 ~50MB
2. **灵活控制**：可动态调整频率、音量、持续时间
3. **淡入淡出**：易于实现平滑的音频切换
4. **离线可用**：无外部音频依赖

### 实现细节
```javascript
// 音效合成示例
playSfx(type) {
    const osc = this.ctx.createOscillator();
    const gain = this.ctx.createGain();
    osc.connect(gain);
    gain.connect(this.ctx.destination);

    switch(type) {
        case 'dice':
            osc.type = 'triangle';
            osc.frequency.linearRampToValueAtTime(600, now + 0.15);
            gain.gain.exponentialRampToValueAtTime(0.001, now + 0.2);
            break;
        // ...
    }
}
```

### 后果
- **正面**：安装包仅 ~120-150MB，无外部音频依赖
- **负面**：不同设备音频效果可能略有差异

## ADR-004: localStorage 持久化

### 状态
已采纳

### 背景
游戏需要保存玩家进度（碎片、成就、结局），支持跨会话继承。

### 决策
使用浏览器 localStorage 存储所有持久化数据。

### 替代方案

| 方案 | 优点 | 缺点 |
|-----|------|------|
| localStorage（采纳） | 简单、无服务器依赖、够用 | 容量有限（~5MB）、同步 |
| IndexedDB | 容量大、异步、支持结构化 | 复杂度高 |
| File System | 容量无限 | 需用户授权、安全限制 |
| Server Backend | 可靠、跨设备同步 | 需服务器、离线不支持 |

### 理由
1. **够用**：游戏数据量小（碎片/成就状态 + 历史记录 < 100KB）
2. **简单**：API 简洁，同步读写足够
3. **离线**：无需服务器，完全离线可用
4. **无依赖**：无需用户登录或网络

### 存储内容
| 键 | 大小估算 | 内容 |
|---|---------|------|
| `cg_fragments` | ~2KB | 碎片解锁状态 |
| `cg_achievements` | ~2KB | 成就解锁状态 |
| `cg_endings` | ~1KB | 结局记录列表 |
| `cg_skipIntro` | ~1B | 跳过介绍标志 |

### 后果
- **正面**：实现简单、离线可用、无服务器成本
- **负面**：无法跨设备同步、清除浏览器数据会丢失进度

## ADR-005: 双桌面框架支持（Tauri + Electron）

### 状态
已采纳

### 背景
项目需要提供桌面应用版本以提升用户体验。

### 决策
同时支持 Tauri（Rust）和 Electron（Node.js）两种桌面打包方案。

### 替代方案

| 方案 | 优点 | 缺点 |
|-----|------|------|
| Tauri + Electron（采纳） | 覆盖多平台、灵活选择 | 两套配置维护 |
| Electron Only | 生态成熟、打包稳定 | 包体积大（~150MB） |
| Tauri Only | 轻量（~10MB）、性能好 | 生态较新 |
| PWA | 无需安装、包小 | 功能限制、桌面体验弱 |

### Tauri 配置
```json
// src-tauri/tauri.conf.json
{
    "build": {
        "distDir": "../dist-web",
        "devPath": "http://localhost:8080"
    },
    "package": {
        "productName": "资本轮回与破局"
    }
}
```

### Electron 配置
```json
// package.json
{
    "name": "marx-history-game",
    "main": "electron/main.js",
    "build": {
        "appId": "com.marx.historygame",
        "productName": "资本轮回与破局",
        "files": ["index.html", "css/**/*", "js/**/*", "yaml/**/*", "assets/**/*", "electron/**/*"]
    }
}
```

### 理由
1. **用户选择**：不同用户有不同偏好
2. **平台覆盖**：Linux 用户更适合 AppImage（Tauri）
3. **风险分散**：任一框架出问题不影响发布
4. **轻量选项**：Tauri 提供更轻量的 Linux 体验

### 后果
- **正面**：覆盖多平台，提供轻量选项
- **负面**：两套打包配置需同步维护

## ADR-006: 纪元主题系统

### 状态
已采纳

### 背景
游戏跨越三个历史纪元，需要视觉和听觉上的差异化体验。

### 决策
通过 CSS 变量和类切换实现主题化，配合差异化背景音乐。

### 实现
```javascript
applyEpochTheme() {
    document.body.className = `epoch-${this.epoch}`;
    this.audio.setEpoch(this.epoch);  // 切换背景音乐
}

// css/themes.css
.epoch-1 { --bg-color: #1a0a0a; --accent: #8b0000; }
.epoch-2 { --bg-color: #0a0a0a; --accent: #4a4a4a; }
.epoch-3 { --bg-color: #0a0a1a; --accent: #0066cc; }
```

### 理由
1. **沉浸感**：不同纪元应有不同的视觉氛围
2. **教育意义**：通过视觉强化历史阶段的认知
3. **技术实现简单**：纯 CSS 即可完成主题切换

### 后果
- **正面**：增强游戏沉浸感和教育效果
- **负面**：需维护多套 CSS 主题变量

## ADR-007: D20 骰子随机性系统

### 状态
已采纳

### 背景
Roguelike 游戏需要随机性增加 replay value，同时体现"历史偶然性"概念。

### 决策
每次选择后触发 D20 骰子，根据结果修正数值：
- 1-5：坏运气（惩罚）
- 6-15：平稳（无修正）
- 16-20：好运气（奖励）

### 理由
1. **游戏性**：不可预测性增加 replay value
2. **教育隐喻**：历史充满偶然性
3. **风险/回报**：玩家需在确定性和随机性间权衡

### 后果
- **正面**：增加游戏 replay value，强化教育隐喻
- **负面**：可能因骰子运气导致挫折感

## ADR-008: 认知碎片 Meta-progression

### 状态
已采纳

### 背景
需要鼓励多周目游玩，同时传递马克思主义理论知识。

### 决策
设计 8 种"认知碎片"，达成特定条件解锁，下周目可携带入场获得加成。

### 碎片列表
| ID | 名称 | 解锁条件 | 效果 |
|---|------|---------|------|
| surplusValue | 剩余价值觉醒 | 达成相关结局 | +5/回合 |
| historicalMaterialism | 唯物史观萌芽 | 唯物史观相关 | -5 初始矛盾 |
| ... | ... | ... | ... |

### 理由
1. **多周目激励**：解锁碎片后有新体验
2. **知识传递**：碎片内容包含理论知识点
3. **策略深度**：不同碎片提供不同游戏风格

### 后果
- **正面**：增加 replay value，传递理论知识
- **负面**：新玩家与老玩家差距可能较大
